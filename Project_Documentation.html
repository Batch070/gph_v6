GPH Automated Fine Payment System

Comprehensive Project Documentation & Architecture

Date: April 2026

1. Project Overview

The GPH Automated Fine Payment System is a comprehensive digital solution designed for Government Polytechnic Hamirpur to automate the student clearance process, attendance tracking, and fine collection. It replaces manual paperwork with a streamlined, multi-role digital workflow.

2. Technology Stack

The application is built using a modern, scalable, and lightweight technology stack to ensure high performance and easy deployment.

Backend (API Server)

  • FastAPI (Python): High-performance web framework for building APIs.
  • Uvicorn: Lightning-fast ASGI server.
  • SQLAlchemy: ORM for database interactions.
  • PyMySQL: Python driver for MySQL/MariaDB.
  • Pydantic: Data validation and settings management.

Frontend (Client-Side)

  • Vanilla HTML5 & CSS3: Pure semantic HTML with responsive CSS Grid/Flexbox architectures. No heavy frameworks (like React/Angular) to keep load times near-zero.
  • Vanilla JavaScript: For DOM manipulation and API calls.
  • FontAwesome 6: Self-hosted local webfonts for vector icons.

Database & Infrastructure

  • MariaDB 10.6: Relational database for robust data persistence.
  • Docker & Docker Compose: Containerization for seamless deployment and environment consistency.

Third-Party Integrations

  • OpenRouter API (AI Vision): Uses Google's Gemma-4-26b-a4b-it vision model for OCR data extraction.
  • Razorpay: Secure payment gateway integration with webhook verification.
  • SMTP: Automated email dispatch for notifications and receipts.

3. System Architecture & Workflows

Role-Based Access Control (RBAC)

The system features secure JWT-based authentication with highly specific role assignments:

  • Admin: Manages global settings, branches, faculty, and views institution-wide analytics.
  • Student: Submits clearance requests, tracks attendance, and pays fines.
  • Class Incharge: Uploads attendance, finalizes fines, and approves departmental clearance.
  • HOD (Head of Department): Oversees department analytics and approves clearance.
  • Specialized Faculty (PTI, Librarian, Canteen, ANO, Hostel Superintendent): Review and approve clearance requests specific to their domain.

AI-Powered Attendance Extraction Workflow

  1. The Class Incharge uploads a photo of the physical attendance register (JPEG/PNG).
  2. The image is sent to the OpenRouter Vision AI Model alongside a strict system prompt.
  3. The AI performs OCR and data extraction, analyzing the columns to find student names, roll numbers, and attended classes.
  4. The AI returns structured JSON data, which the backend parses, validates against the database, and presents to the Incharge for review.
  5. The Incharge confirms the data, saving it to the database.

Automated Fine Calculation Workflow

Once all subject attendances are uploaded for a semester, the Class Incharge clicks "Finalize Attendance". This triggers a background task that:

  • Aggregates total theory and practical classes for every student.
  • Calculates the attendance percentage.
  • Applies the institutional fine rules (e.g., if attendance is below 75%, a specific fine amount is levied).
  • Generates a Fine record in the database.
  • Sends an automated email report to the student detailing their attendance percentages and fine amount.

The Clearance & Payment Lifecycle

  1. Initialization: A student logs into their dashboard. The system checks if their fine and attendance data has been uploaded.
  2. Request Submission: The student clicks "Request Clearance". The system automatically generates pending requests for all mandatory roles (HOD, Incharge, PTI, Librarian, Canteen) and conditional roles (ANO if NCC cadet, Hostel Superintendent if hosteller).
  3. Faculty Review: Faculty members log into their respective portals, see pending requests, and mark them as "Approved" or "Rejected" (with notes).
  4. Payment Unlock: Once all pending requests are approved, the payment gateway is unlocked on the student's dashboard.
  5. Transaction: The student pays via Razorpay. The backend creates an order, and the frontend opens the Razorpay checkout widget.
  6. Secure Verification: Razorpay fires a Webhook to the backend upon successful payment. The backend cryptographically verifies the webhook signature.
  7. Clearance: The student's fine status updates to "Paid", their overall status updates to "Cleared", and an HTML receipt is generated and emailed.

4. Security Measures

  • Password Cryptography: Passwords are hashed using bcrypt with auto-generated salts.
  • Stateless Authentication: JSON Web Tokens (JWT) are used for secure, stateless session management.
  • Webhook Verification: Payment webhooks use HMAC-SHA256 signatures to prevent spoofing.
  • Security Headers: Middleware injects strict Content-Security-Policy (CSP), X-Frame-Options, and XSS Protection headers.
  • Rate Limiting: slowapi restricts login attempts to prevent brute-force attacks (e.g., 60 requests/minute limit).
  • SQL Injection Prevention: SQLAlchemy ORM uses parameterized queries exclusively.

5. Database Schema Highlights

Table Name Purpose
students Stores student demographics, branch, semester, boolean flags (NCC, Hosteller), and aggregated attendance percentages.
faculty Stores faculty credentials and specific roles (Enum: Admin, HOD, PTI, Librarian, etc.).
subject_attendance Stores granular attendance data per subject, group type (Theory/Lab), and session mappings.
fines Links to students; tracks fine amount, semester, payment status (Paid/Unpaid), and Razorpay transaction IDs.
requests The core clearance ledger. Links a student to a faculty member, tracking Approval/Rejection status and notes.
